# Reading Experience Vision Draft

> 文档状态：草案 / 初步想法（未立项）
> 最近更新：2026-06-25
> 归属：跨 `reader-runtime-web` / `apps/editor` / `packages/language` 的阅读体验愿景
> 相关：`presentation-modes-and-capability-layering-draft.md`（能力分层）、`../apps/editor-dip-effect-library.md`（DIP 滤镜库）、`packages/reader-runtime-web.md`（阅读器边界）

## 0. 这份文档要回答什么

KMD 一直被当作「会动的排版」来建设——解析、布局、effect、镜头都围绕「字怎么呈现、怎么动」。但有一个维度此前没有任何规划文档承载：**人到底是在什么环境里「看」一篇 KMD 的**。

它最早作为纯阅读器（reader-runtime）被设想，可阅读体验本身的愿景从未落纸。直到 DIP 滤镜库审查时撞到一个问题——背景图。一旦引入背景图作为观看体验的一部分，就立刻牵出「背景的亮度和模糊」这种**可读性**需求；而这批操作恰好是 DIP 滤镜库里那些「作用在单字上会退化、需要连续色调表面」的算子的天然归宿。

所以这份草案立三件事：

1. 给「KMD 阅读体验」一个愿景锚点（此前只有 `knowledge/language/design.md:99` 一句话）。
2. **分场景**初步规划：字幕 / 提词器 / 阅读器 / 叙事作品——当前主线是**叙事作品**。
3. 把**背景定为一等表面**，并接清三条缝：与能力分层、与 DIP 滤镜库、与镜头 `frame` 层的关系。

> 本文是愿景与初步规划，不是近期实施计划。立项触发条件见 §7。

## 1. 愿景

> **KMD 让作者在「可选的图像环境」之上，呈现可读、有设计感的动态排版。**

拆两个关键词：

- **可读（legible）优先**：无论背景是纯色、照片还是视频，文字必须始终可读。可读性不是事后调色，是阅读体验的**地基约束**——背景的存在不能以牺牲文字为代价。
- **有设计感（designed）**：在可读的前提下，排版、节奏、氛围共同服务于「这篇内容想给人的感受」。design.md 的原话「用 kmd 表达富有设计感的特殊排版」在此延伸到「连同它的观看环境一起被设计」。

这条愿景跨形态成立——它不绑定叙事。但**不同场景对「可读」与「设计感」的配比要求差别很大**，所以要分场景谈。

## 2. 多场景分层（当前主线：叙事作品）

KMD 不止一种用法。把阅读体验按场景拆开，每个场景对背景、可读性、动效的诉求不同：

| 场景 | 阅读姿态 | 背景诉求 | 可读性底线 | 动效/氛围 | 现状 |
|---|---|---|---|---|---|
| **叙事作品**（主线） | 沉浸、被引导、线性推进 | **强**：背景是叙事的一部分（场景、情绪、隐喻） | 文字须在任意背景上可读 → 压暗/虚化/底衬 | 重：entrance + behavior + 镜头 | 在建 |
| 阅读器（长文/电子书） | 自主、可回看、长时间 | 中：护眼底色、轻氛围图，不喧宾夺主 | 高：长时间阅读，对比度/疲劳敏感 | 轻：克制的揭示，避免干扰 | reader-runtime 在建 |
| 字幕 / 歌词 | 瞥视、伴随主画面 | 弱：通常叠在视频/音乐可视化上 | **极高**：半透明底衬/描边保证在任意画面上可读 | 中：强调、节奏卡点 | 潜在 |
| 提词器 | 扫读、匀速滚动 | 几乎无：纯色或暗底 | 高：大字号、强对比、低干扰 | 极轻：仅滚动/高亮 | 潜在 |

**当前主线是叙事作品**，因此本草案的细节优先服务它；但背景表面的设计要刻意保留跨场景普适性（§3），避免把阅读体验锁进叙事。

叙事场景的典型诉求，正是它把「背景」从可有可无推到一等公民：

- 一张场景照片/插画作背景，文字浮于其上 → 必须**压暗或虚化背景**，否则文字淹没在高频细节里。
- 背景随叙事推进切换、明暗呼吸 → 背景的亮度/模糊本身是**可被时序驱动的表现量**。
- 「文字踩进水里」这类镜头 → 背景（水面）与文字（下沉）是两个被分别处理的表面。

## 3. 背景作为一等表面

本草案的核心架构主张：

> **`background` 是独立于「逐字字形」与「镜头 frame」的第三类表面。它的处理操作首要为可读性，其次为氛围。**

### 3.1 为什么独立成一等表面

- 它不是「某个字的滤镜」——背景是整幅、连续色调的，作用对象与逐字小纹理根本不同。
- 它也不是「镜头 frame 后处理」——`frame` 是相机合成**整帧**后的叙事后处理（镜头层、形态专属）；而背景作为**阅读基底**是跨形态普适的（字幕、提词器、阅读器都要「字铺在某种底上且可读」）。两者不同层（§4.3）。
- 它承载一组明确的操作，且这些操作有**可读性语义**而非纯装饰：

| 操作 | 可读性作用 | 氛围作用 | DIP 对应 |
|---|---|---|---|
| `bg.brightness` / dim | 压暗背景抬升文字对比度（等价字幕的半透明黑条/scrim） | 情绪、昼夜 | 点运算（亮度） |
| `bg.blur` | 虚化高频细节，让文字从背景「浮」出来 | 景深、梦境 | 高斯卷积 |
| `bg.tint` / duotone | 统一色调，避免杂色干扰文字 | 整体氛围 | 点运算 / 色彩映射 |
| `bg.vignette` | 边缘压暗，向心聚焦文字 | 老照片、聚光 | 径向点运算 |

### 3.2 这正好是 DIP 连续色调滤镜的归宿

DIP 滤镜库审查暴露：blur/brightness/duotone/bloom/halftone/edge 这类算子作用在**逐字小纹理**上会退化（细节见 `../apps/editor-dip-effect-library.md` §8.1、spec §0.3）。它们的教科书级正确对象是**一张真实的连续色调照片**——背景图就是。

因此背景表面接线后，DIP 库这批滤镜**复用同一个 `fn`、零改动**即可作用其上（兑现 DIP spec §1.1「写一次、多作用域复用」）。两条线在此合流：阅读体验需要「背景可读性处理」，DIP 库需要「连续色调作用对象」，背景表面同时满足两者。

### 3.3 「可读性」作为地基约束，不只是一个效果

值得强调的设计姿态：背景压暗/虚化在叙事/字幕里**不是作者随手加的滤镜，而是系统应保证的可读性底线**。可以考虑（开放，§6）：

- 一个**自动可读性保障**：当检测到背景与文字对比度不足时，自动施加最小必要的 scrim/dim（类似字幕渲染器的自动描边/底衬），作者可覆盖但默认有保底。
- 把 `bg.*` 既暴露为**作者可控参数**，又作为**系统可读性策略**的底层原语——二者同一套实现，不同默认。

## 4. 三条缝的接法

### 4.1 与能力分层（`presentation-modes-and-capability-layering-draft.md`）

在能力分层表里新增一层：**「背景作阅读基底」属阅读核心、跨形态普适**（已回写 DIP 概览 §9.2）。与之相对，「背景作镜头场景」（叙事换景、运镜天幕）属形态专属的镜头层。同一张背景图，两种身份分属两层——切分点是「它在为可读性服务，还是为叙事运镜服务」。

### 4.2 与 DIP 滤镜库

背景表面是 DIP 库连续色调滤镜的作用对象（§3.2）。落地时，`bg.blur` 等很可能就是 DIP 库 `blur`/`duotone`/`vignette` 的 `fn` 路由到背景容器，而非另写一套。DIP spec §7.3 已记此联动。

### 4.3 与镜头 `frame` 作用域（DIP spec §7.1）

| | `background`（本草案） | `frame`（镜头层） |
|---|---|---|
| 作用对象 | 背景图层（文字之**下**） | 相机合成后的**整帧**（含文字） |
| 首要目的 | 可读性（其次氛围） | 叙事后处理（整屏 CRT/水下/做旧） |
| 普适性 | 跨形态（阅读核心） | 形态专属（镜头能力） |
| 与相机 | 通常随相机变换 | 相机变换**之后** |

两者都「不是逐字」，但分属阅读层与镜头层，不要合并。

## 5. 与现有模式（stage / scroll / page）的交互

背景在三种呈现模式下的行为待定，但初步直觉：

- `stage`（定比例信箱）：背景填满信箱，随场景切换/镜头移动，最贴近叙事。
- `scroll`（滚动）：背景可固定（视差）或随内容滚动；长文/阅读器场景。
- `page`（分页）：背景按页切换，每页一底。

背景写在 frontmatter（全篇默认）还是可逐段/逐场景覆盖，是核心开放项（§6）。

## 6. 开放项

- **背景的声明位置**：frontmatter 全局默认 vs 逐段/逐场景覆盖 vs 随 `scene.clear` 切换。
- **可读性保障是否自动**：系统自动 scrim/dim（有保底）vs 纯作者手控。倾向「有保底 + 可覆盖」，但需实测对比度阈值。
- **背景与 `scene.clear` 的交互**：`---` 清场时背景是否一起淡出/保留/独立切换。
- **三模式行为**（§5）定稿。
- **视频/动态背景**是否纳入，还是先只做静态图 + 可时序驱动的 `bg.*` 参数。
- **`bg.*` 实现归属**：复用 DIP 库 fn 路由到背景容器（倾向）vs 背景独立子系统。
- **复用既有 `backgroundLayer` 基底**：现实是 `StageManager.world` 下已有一个 `backgroundLayer` 容器（`StageManager.ts:25/37/41`），但**目前只挂纯色**（`setBackgroundColor`，color-only），**无背景图、无 `bg` KMD 命令**——`stagePresets.ts` 里查无 `bg`（CLAUDE.md 称「`bg` registered in stagePresets」是过时/愿景，须一并订正）。即：阅读基底的**承载容器已存在但未被用于图像**。本草案的背景图表面很可能就落在这个 `backgroundLayer` 上，`bg.*` 处理作用其上——但要先确认它与未来镜头层背景（叙事换景）是同一容器还是要分层。

## 7. 立项触发条件

满足任意两条，可把本草案升级为正式规划：

- 叙事 demo 实际需要背景图 + 文字可读性处理（DIP M2/M3 demo 阶段很可能触发）。
- reader-runtime 的阅读 UI 推进到需要护眼底色/背景策略。
- 出现字幕/提词器场景的真实需求，需要跨场景统一背景/可读性模型。
