# Presentation Modes & Capability Layering Draft

> 文档状态：草案 / 初步想法（未实施，未立项）
> 最近更新：2026-06-13
> 触发来源：DIP 滤镜库规划期间暴露的 “stage” 命名碰撞与作用域归属问题

## 1. 为什么单独放这里

这是一条跨包跨应用的生态级判断，最初在 `apps/` 下的 DIP 滤镜库规划（`../apps/editor-dip-effect-library.md` §9）里被推导出来，但它比滤镜库本身大得多：它关于 **KMD 作为多形态输出引擎时，能力该如何分层、镜头系统该去哪、命名该怎么解碰撞**。埋在一个 app 子项目里以后无人可寻，故抽到生态目录留作初步想法。

本文件**不是近期实施计划**，也不改变现有代码。它记录方向，等到“下次碰 stage / mode 矩阵成熟 / `packages/core` 抽取”任一时机来临时，作为立项依据。

## 2. KMD 是多形态的，不只是叙事

KMD 的潜在输出形态至少有：**视频素材、字幕、提词器、叙事作品**（以及未来未知的）。这决定了一个设计风险：若把“偏某一形态”的能力做成常驻默认开启，KMD 的默认身份就会被那一形态绑架。

当前最明显的例子是**镜头/电影感系统**（`StageRuntime`/`StageManager`：cam.move/zoom、bg、scene.clear）。它服务叙事/视频，但字幕和提词器**主动排斥**会平移缩放的相机——它们要的是稳定可读、流式滚动。

## 3. 能力分层原则：普适 → 核心，形态专属 → 能力模块

按能力对各输出形态的**普适性**分层：

| 层 | 普适性 | 字幕 | 提词器 | 视频素材 | 叙事 | 归属 |
|---|---|---|---|---|---|---|
| 内容 + 时序（文字揭示） | 全普适 | ✓ | ✓ | ✓ | ✓ | 核心 |
| 逐元素动效（effect：behavior/entrance/filter） | 大体普适、按需 | 可选 | 罕用 | 大量 | 大量 | 核心 |
| 镜头/电影感（cam/bg/scene） | **形态专属** | ✗ | ✗ | 部分 | ✓ | **能力模块（待门控/插件化）** |

原则：**普适能力留核心、默认开启；形态专属能力做成可门控的能力模块，按 `mode` 或作品声明启用。** 这样 KMD 的核心身份是“动态文字演出引擎”，而“电影叙事”只是它的一种装配。

## 4. “stage” 三重命名碰撞

`stage` 当前同时指三样东西，是认知负担的来源：

| 名字 | 指什么 | 处置 |
|---|---|---|
| `mode: "stage"` | **呈现模式**（定比例信箱，对 scroll/page） | 既有语言资产，保留 |
| `StageRuntime` / `StageManager` / `stagePresets` | **镜头/电影感子系统** | 既有代码，短期不改名；长期定位见 §5 |
| （DIP 滤镜的全屏作用域） | 已避让，命名为 **`frame`** | DIP 库已落地，不复用 stage |

长期若镜头系统门控/插件化（§5），其命名可顺势从 “stage” 迁往更准确的 “cinematics / 镜头”，把 “stage” 词义收敛回**呈现模式**单一含义。这是迁移的附带收益，不是迁移的理由。

## 5. 镜头系统去向：概念插件，物理三段（对齐 repository-strategy）

`repository-strategy.md` 明确：`packages/core` API 未稳定前不拆物理包。故镜头能力的演进分三段，**不可跳级**：

1. **现在（仅文档）**：命名这条缝——本文件。声明镜头为形态专属能力、内容/动效为普适核心。代码不动。
2. **能力注册缝（下次碰 stage / mode 矩阵成熟时）**：在 core 内让 cam/bg/scene 作为“可门控能力”注册，非叙事 mode（scroll/字幕/提词器）可不加载。手法对标 reader-runtime 已在做的“门控 editor-only 能力”。DIP 滤镜的 `frame` 作用域（`../apps/editor-dip-effect-library-spec.md` §7.1）随这层一起接线——因为 `frame` 是相机合成后的后处理，属镜头层。
3. **物理插件（由 `packages/core` 抽取门槛触发）**：仅在 core API 稳定后，镜头能力沉淀为独立 `@kmd/*` 插件包。

## 6. 开放问题

- `mode` 与“能力集”的映射要不要显式化？（例如 `mode: subtitle/teleprompter` 各预声明启用哪些能力模块）——可能需要扩展 `mode` 取值或引入独立的 capability 声明。
- 门控的粒度：按 `mode` 整体门控，还是按单条命令的 capability 标签门控？
- 镜头能力门控后，reader-runtime 的可移植边界是否因此更干净（叙事/非叙事 runtime 可裁剪）？
- 是否需要在 `knowledge/architecture/` 沉淀“能力分层”为正式架构原则——待本草案被任一实施阶段验证后再升格，现在保持草案。
