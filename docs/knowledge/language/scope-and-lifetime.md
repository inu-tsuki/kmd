# 作用域与生命周期

> 最近更新：2026-07-08
> 状态：封盘（规范定稿 / 未实现）
> 决议依据：D1、D2、D19–D22（见 `design.md` 封盘清单）

本章定义名字如何解析、活多久，以及选项（frontmatter / 块选项）的级联模型。

## 作用域链

链头与表达式里的裸标识符按一条链查找：

```
裸标识符
  → 场景级用户定义（围栏名、@ name = ...）
  → 文档级定义（var 作用域，含 frontmatter var: 表）
  → 内建主语（cam、flow、var …）
  → 特效/指令名（effect / style / layout / stage 注册表）
```

- 正文里裸文字永远是文字；进表达式必须过 `{=` 这道门（见 `control-flow.md`）——门内裸名，门外纯文。
- 链头查找失败不是 parse error，走诊断总线给出建议（doesNotUnderstand 风格）。

## 两级作用域

只有两级，不再多：

| 级 | 声明方式 | 生存域 |
| --- | --- | --- |
| 文档级 | frontmatter `var:` 表；任何位置 `@ var.name = ...` | 全文档（跨场景） |
| 场景级 | `@ name = ...`、围栏名 | 定义处到所属场景的 `---` |

- **`var` 是文档级作用域的名字**。`var.x` 是全限定访问（消歧或明示时用，平时不强制）；**向 `var` 写入即随处声明文档级变量**。frontmatter 的 `var:` 表是文档级初值的声明糖。
- `var.` 前缀因此成为**生命周期的视觉标记**：跨场景的东西必须挂 `var`，挂了 `var` 就必然跨场景。

## 生命周期 = 脚本区间

> 对象的生存域是**脚本文本上的一段区间**，不是播放时间上的一段。

播放状态是脚本位置的纯函数（场景烘焙 / snapshot / seekTo 的既有哲学）。因此：

- 回翻、跳转到区间内任何位置，对象就在，状态由该位置重建——页模式作品翻回来对象不会丢。
- `---` 清除的不是"运行时对象"，是声明"这个名字的场景级作用域到此为止"。
- 悬空引用（场景外引用场景级名字）走诊断。

## 全局命名冲突律

- 禁 shadowing：用户名与内建主语、指令名、其他用户名冲突，一律诊断报错，不静默遮蔽。
- 指令注册表之间（style / effect / layout / stage）重名同样是诊断——废除现行"按查找顺序优先级静默裁决"（`commandCatalog.getFamily` 的 style→effect→layout→stage 顺序）。

## 选项级联

frontmatter 和 `[key=value]` 是**同一张选项表的两个书写位置**，就近覆盖：

| 层 | 位置 | 作用域 |
| --- | --- | --- |
| 引擎默认 | — | — |
| 文档级 | frontmatter（`mode`、`speed`、`designWidth` …） | 全文档 |
| 段级 | `[key=value]` | 仅所修饰的段，段结束即失效（**不粘滞**） |

- 读取规则：段级 → 文档级 → 引擎默认，就近取值。
- **引擎启动键**（`mode`、`designWidth`、`designHeight` 等）文档级专属，段级出现走诊断（"此选项不可段级覆盖"）。
- 想要"从这里开始一直生效"的粘滞语义？那是变量的事（`@ var.x = ...`），不是选项的事——选项保持无状态。

`[...]` 的完整定义：**段级选项（key=value） + 段首句子**（任意完整主谓句：`cam.zoom(...)!`、`[if ...]`、省略主语的段级谓语 `.glitch`）。每个语法位置只承载一种语义强度。

## 实现落差

- `KMDCommandParser.autoConvert` 的 `var.` 字符串直通特判由作用域链取代。
- `KMDParser.validate()` 的已知指令名检查升级为作用域链查找 + 诊断。
- 段级选项目前散写在 `applyBlockOptionCommands` 的 `(blockOptions as any)[key] = value`，需按级联模型收编。
