# ADR/提案模板

> 最近更新：2026-07-07
> 双重用途：
> 1. **事前提案**——重要改动（不可逆、跨子系统、引入新模式/新依赖）在动手前按此格式提交给决策者审阅；
> 2. **事后归档**——被采纳的提案补上"结果"一节，以 `YYYY-MM-DD-short-decision-title.md` 存入本目录，成为 ADR。
>
> 决策者不需要读懂实现，只需核对本模板各栏是否自洽。配套判据见
> `docs/knowledge/architecture/design-patterns-map.md` 的一致性检查清单。

---

```markdown
# 提案：<一句话说清要做什么>

- 日期：YYYY-MM-DD
- 状态：提案中 | 已采纳 | 已否决 | 已被 <链接> 取代

## 回应的力
这个问题为什么现在必须解决？不解决会发生什么？（答不上来 = 可能是过度设计）

## 方案对比
### 方案 A（最简可行）
做法：…
代价 / 局限：…

### 方案 B（推荐）
做法：…
代价 / 局限：…
为什么最简方案不够用：…（这一行是必填项）

## 触碰的不变量
逐条列出：seek 幂等？core/UI 边界？scene.clear 单路径？
参照 `docs/knowledge/runtime/core/lifecycle-invariants.md`。
守护这些不变量的测试在哪？（没有 → 本提案需包含补测试）

## 与 house style 的关系
沿用了 `design-patterns-map.md` 中哪个既有模式？
若引入新模式/新依赖/新全局状态：为什么既有模式不够？

## 可逆性
如果这个决定错了，退回来要多久？改动是否会锁死文件格式、协议或公开 API？

## 结果（归档时补写）
实际落地情况、后续复核条件、相关 commit / PR。
```

---

## 决策者速查：红旗清单

见到以下任一条，暂停并要求补充论证（不一定错，但错误率高）：

- 🚩 新增全局单例或 `globalThis` 挂载
- 🚩 同一件事出现第二种做法（第二条路径、第二个入口）
- 🚩 方案里出现"暂时先""以后再统一""为兼容先复制一份"却没有偿还死线
- 🚩 一个"小需求"的 diff 触碰 8 个以上文件
- 🚩 "这个没法写测试"（几乎总意味着耦合出了问题）
- 🚩 引入新依赖解决 50 行代码能解决的问题
